# i18n

Two layers:

1. **UI translation** â€” admin chrome, field labels, menu titles.
2. **Content translation** â€” Eloquent model fields stored per-locale
   (via `dskripchenko/laravel-translatable`).

## Locale resolution

`LocaleResolver` picks the locale via 5-step priority:

1. `?_locale=en` query param (explicit override)
2. `X-Admin-Locale` header (set by SPA after user changes locale)
3. `users.locale` column (if present)
4. `XSRF-LOCALE` cookie
5. `Accept-Language` header
6. `config('admin.ui.default_locale', 'ru')`

Selected locale is applied via `app()->setLocale()` and exposed in
`Manifest.locale`.

## UI language packs

Frontend stores translation keys in `i18nStore`. Backend ships
`resources/lang/{en,ru}/admin.php` with all string keys used by the
admin shell. Override / extend in your host:

```php
// config/admin.php
'i18n' => [
    'override_path' => resource_path('lang/admin'),
],
```

Add a missing locale:

```php
// resources/lang/de/admin.php
return [
    'menu.users' => 'Benutzer',
    'menu.settings' => 'Einstellungen',
    // ...
];
```

## SPA locale change

```ts
const locale = useLocaleStore()
await locale.setLocale('de')
```

Sends POST `/api/admin/system/setLocale {locale: 'de'}`. Server saves to
user.locale (if column exists) and sets cookie. SPA reloads i18n bag.

## Content translation (models)

Use `dskripchenko/laravel-translatable`:

```php
use Dskripchenko\LaravelTranslatable\Traits\TranslationTrait;

class Article extends Model
{
    use TranslationTrait;
    protected array $translatable = ['title', 'body', 'excerpt'];
}
```

Reading:

```php
$article->t('title');                  // current locale
$article->t('title', 'de');            // explicit
Article::whereTranslation('title', 'Hi')->get();
Article::orderByTranslation('title')->get();
```

## TranslatableField

The admin's `TranslatableInput` renders a tabbed input â€” one tab per
locale:

```php
TranslatableInput::make('title')->locales(['en', 'ru', 'de']),
```

Stored as a JSON map; the bridge layer (`TranslatableFieldBridge`) wires
read/write back to the underlying `translations` polymorphic table.

## Adding a locale

1. Create `resources/lang/{locale}/admin.php` (override only what you
   change; missing keys fall back to default locale).
2. Add the locale to `config('admin.ui.locales')`:
   ```php
   'ui' => [
       'default_locale' => 'en',
       'locales' => ['en', 'ru', 'de', 'zh'],
   ],
   ```
3. The locale-switcher in the topbar populates from this list.

## Best practices

- Keep translation keys flat-namespaced (`menu.users`, not nested).
- Don't translate code identifiers (slugs, role keys, settings keys).
- Date/number formats â€” use ICU via `Intl` (`new Intl.DateTimeFormat`
  on frontend), don't hand-roll.
- For long-form translatable content, prefer `TranslatableInput` over
  multiple plain Input fields per locale.
